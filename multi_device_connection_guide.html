순수 Native C++ 멀티 디바이스(다중 장비) 접속 및 제어 가이드

본 문서는 C++ 개발자가 본 라이브러리(CommunicationWrapper.dll)를 사용하여 두 대 이상의 Power PMAC 장비에 동시에 접속하고 제어하는 방법에 집중하여 설명합니다.

1. 다중 장비 접속(Connect) 원리

이전 구조와 달리, 이제 UI 프로젝트는 /clr 옵션이 필요 없는 순수 Native C++ 로 동작합니다.
내부적으로 DLL에서 메모리 포인터 형태의 Handle(void*)을 발급받아, 이를 캡슐화한 PmacNativeManager 클래스를 통해 IP 주소별로 다수의 장비를 완벽히 격리하여 관리합니다.

접속 예제 코드 (Native C++)

#include "PmacNativeManager.h"

// 1. 매니저 객체 선언 (보통 다이얼로그의 멤버 변수로 선언)
PmacNativeManager m_manager;

// 2. IP 문자열 정의 (표준 std::string 사용)
std::string ipA = "192.168.0.200";
std::string ipB = "192.168.0.201";

// 3. 각각의 IP로 물리적 접속 시도 (AddDevice는 내부에서 자동 호출됨)
// 주의: 응답성이 중요한 경우 UI 멈춤 방지를 위해 std::thread 등 비동기로 호출 권장
bool isConnectedA = m_manager.Connect(ipA);
bool isConnectedB = m_manager.Connect(ipB);

if (isConnectedA && isConnectedB) {
    // 다중 장비 동시 접속 완료 (각 장비는 독립된 소켓과 세션을 유지함)
}

2. 다중 장비 개별 제어 (Command)

접속이 완료되면 매니저의 SendCommand 함수에 해당 장비의 IP와 명령어를 전달하기만 하면 됩니다. DLL 내부에서 해당 IP의 Handle을 찾아 독립된 통신 객체로 명령을 전송하므로 장비 간 패킷 충돌이나 뻗음(Crash) 현상이 전혀 발생하지 않습니다.

명령어 전송 예제 (Native C++)

// PMAC A 장비(192.168.0.200)의 1번 모터 조그(JOG) 구동
std::string resA = m_manager.SendCommand("192.168.0.200", "#1J+");

// PMAC B 장비(192.168.0.201)의 1번 모터 조그(JOG) 구동
std::string resB = m_manager.SendCommand("192.168.0.201", "#1J+");

// PMAC A 장비 정지
m_manager.SendCommand("192.168.0.200", "#1J/");

3. 실시간 동시 모니터링 (타이머 기반)

여러 장비의 상태(위치값 등)를 동시에 모니터링할 때는 C++의 타이머(OnTimer)를 사용하여 SendCommand를 연속 호출하면 됩니다. 더 이상 장비를 전환할 때마다 Re-connect(끊고 다시 연결)를 수행할 필요가 없습니다.

위치값 병렬 폴링 예제 (MFC)

void CPDKSampleUIDlg::OnTimer(UINT_PTR nIDEvent)
{
    // 타이머 틱(예: 200ms)마다 양쪽 장비에 상태를 요청
    
    // PMAC A 위치 쿼리
    if (m_manager.IsConnected("192.168.0.200"))
    {
        std::string resA = m_manager.SendCommand("192.168.0.200", "Motor[1].Pos");
        // 응답 예시: "Motor[1].Pos=123.456" 
    }

    // PMAC B 위치 쿼리
    if (m_manager.IsConnected("192.168.0.201"))
    {
        std::string resB = m_manager.SendCommand("192.168.0.201", "Motor[1].Pos");
    }

    CDialogEx::OnTimer(nIDEvent);
}
[!IMPORTANT]
m_manager.SendCommand를 호출하기 전에 응답 지연(Timeout)을 방지하기 위해 m_manager.IsConnected()로 연결 상태를 먼저 체크하는 것을 권장합니다.
